# Post-change product review

## Findings

| Severity | Category | Evidence | Required fix | Confidence |
|---|---|---|---|---|
| high | MVP scope | The MVP includes a versioned event schema, cross-process locking, fsync and torn-write handling, a hostile-output sanitizer, reflection taxonomy, three reports, recovery notes, and purge. T1 alone has six detailed acceptance criteria; the plan has three implementation waves and three validation gates. This is a local operator journal, not a concurrent service or shared evidence system. | Reduce the MVP to recording one allowlisted command invocation and its exit status with bounded sanitized output, then rendering a simple command/failure report. Defer locking durability edge cases, classification taxonomy, recovery notes, and purge until real use demonstrates a need. Keep path containment and redaction appropriate to the retained artifact format. | high - directly evident from the planned command surface and task breakdown. |
| high | prerequisite justification | The plan makes CPython 3.13 a new local prerequisite and adds a host selector solely for this opt-in journal. The stated benefit is avoiding nested Docker, but the plan provides no evidence that the journal requires 3.13-specific functionality or that an already available supported host Python cannot run it. | Remove the exact CPython 3.13 prerequisite unless a concrete required capability is identified and verified. Use a supported existing host Python version range, or make the journal run through the existing tooling boundary if that preserves the required workflow. Document only the minimum verified prerequisite. | high - the plan specifies standard-library implementation but gives no 3.13-specific requirement. |
| medium | workflow simplicity | `note --kind recovery --command-id --service` introduces a second, manual representation of recovery without executing or verifying it. The plan already states it is always unverified and cannot change workflow state, so it adds operator input and explanation burden without completing a journal need. | Remove `note` from the MVP. Let the journal report observed commands and failures only. Add operator annotations only after a concrete review use case establishes what information must be captured and how it will be used. | high - the plan expressly labels the feature display-only and unverified. |
| medium | framework excess | `purge` adds a destructive command, canonical containment logic, symlink/junction defenses, and dedicated tests even though automatic retention cleanup is deferred and artifacts are already isolated under ignored `.tmp/update-runs/`. This makes deletion behavior a first-class feature before the journal's core workflow is proven. | Defer `purge` and its implementation/test/documentation surface. State the artifact location and let existing operator-controlled local cleanup handle it until retention requirements are known. | high - the plan calls purge an MVP command while explicitly deferring retention cleanup. |
| medium | operator boundaries | The journal adds a new `start`, explicit episode-ID transfer, `run`, and `reflect` sequence around every existing command. While `run` correctly avoids automatic advancement, the plan does not show that cross-command correlation is required for the first usable journal, nor does it make the added ceremony optional for a single failed command. | Make a single `run update|validate|plan` create and print an episode by default, with `--episode` as an optional join mechanism when an operator deliberately wants correlation. Keep automatic advancement prohibited. This preserves the existing review boundary with less routine workflow overhead. | medium - explicit joining is a reasonable correlation design, but its mandatory use is not justified for the smallest useful journal. |
